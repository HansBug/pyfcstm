# `model1.xml` 与 `model1_fixed_v2.xml` 的模型层差异说明

## 1. 这次修改的定位

这次不是把原模型整体重画了一遍，而是在 **保持整体状态机骨架基本不变** 的前提下，补齐并校正了几处关键模型语义，使状态机条件、场景时序和触发信号三者能够对得上。

如果只从模型层面总结，`v2` 的变化可以归纳为一句话：

> 把原版里两处“条件写了，但条件依赖的量没有在场景里被清楚表达”的地方，改成了可以直接观察、直接解释的阈值条件；同时把几条本应作为控制器输入的触发信号方向纠正过来。

## 2. 哪些内容没有变

这些内容在原版和 `v2` 之间保持不变：

- 顶层主流程仍然是：`Idle` 在收到 `Sig1` 后进入 `Control`
- `Control` 仍然是同一个复合状态，内部仍然分成 4 个并行区域
- 各区域的状态集合没有重构，仍然是图上那几组状态
- 主要事件名字没有换，仍然围绕 `Sig1` 到 `Sig18`
- 大部分状态迁移链路没有改，尤其是 `Sig2`、`Sig5`、`Sig6`、`Sig4` 等驱动的主链路仍保留

换句话说，`v2` 不是“换了一套业务逻辑”，而是对原模型做了 **条件语义澄清** 和 **场景一致性修正**。

## 3. `v2` 的核心模型修改

### 3.1 把 `A -> B` 的转移条件，从抽象比较改成了可观测阈值

原版中，`Control` 第 1 个并行区域左上角那条从 `A` 到 `B` 的条件迁移，图上写的是：

- `when(a<b)`
- 或者等价地说，这条迁移依赖“`a` 小于 `b`”

但从当前样例的整体模型看，`a`、`b` 并不是这份场景里被明确展示、明确演化的观测量。对外沟通时，会很难回答两个问题：

- 到底是谁在改变 `a`、`b`
- 场景进行到哪一步时，这个条件应该成立

`v2` 把这条条件改成了：

- `y < 2200`

这代表模型语义从“两个抽象量之间的大小关系”改成了“一个可观测量是否低于明确阈值”。

这样改之后，这条迁移就能直接和场景中的数值点对应起来：

- 当 `y = 2300` 时，条件不满足，状态仍停留在 `A`
- 当 `y = 2099` 时，条件满足，于是触发 `A -> B`

从模型角度看，这个修改的意义是：

- 原版是“写了条件，但条件和场景没有清楚绑定”
- `v2` 是“把条件绑定到场景中已经出现的实际观测量 `y` 上”

### 3.2 把 `F -> W` 的转移条件，从抽象比较改成了新的高度阈值条件

原版中，`Control` 第 2 个并行区域左侧那条从 `F` 到 `W` 的条件迁移，图上写的是：

- `when(c<d)`

这和上面的问题类似。`c`、`d` 在原图和原场景里并没有被作为清晰的业务量展示出来，因此这条条件在模型解释上是悬空的。

`v2` 把这条条件改成了：

- `z < 1200`

这意味着第 2 个并行区域的进入条件被重新定义为：

- 不是“某两个抽象内部量比较”
- 而是“一个明确量 `z` 下降到阈值 `1200` 以下”

从模型语义上看，这个修改把第 2 个并行区域的触发依据变成了可以单独解释的业务规则。

### 3.3 为第 2 个并行区域新增了独立观测量 `z`

为了支撑上一条修改，`v2` 不只是把条件文字改掉，而是把 `z` 真正补进了模型语义里。

相对于原版，`v2` 新增了一个独立量：

- `z`

并且给它补上了明确的场景取值轨迹：

- `z = 2000`
- `z = 1800`
- `z = 1300`
- `z = 1100`

这组数值的作用不是装饰，而是给第 2 个并行区域的条件迁移提供“何时应该触发”的依据：

- `z = 2000`、`z = 1800`、`z = 1300` 时，`z < 1200` 还不成立
- 到 `z = 1100` 时，`z < 1200` 才成立，因此 `F -> W` 才有触发依据

因此，`v2` 在模型层面新增的不是一个孤立变量，而是：

- 一个新的可观测状态量
- 一条围绕该状态量的明确阈值规则
- 一组与场景时序对齐的观测点

这使得第 2 个并行区域的行为，终于能像第 1 个并行区域那样，被场景清楚解释。

### 3.4 把 `Sig9`、`Sig8`、`Sig7` 的场景方向改成“模块触发控制”

结合原版状态机图和时序图可以看到：

- `Sig9` 触发的是 `H` 内部 `L -> M`
- `Sig8` 触发的是第 2 个并行区域 `H -> G`
- `Sig7` 触发的是第 3 个并行区域 `S -> O`

这三条迁移在状态机语义上，都是“控制状态机因为收到某个信号而发生切换”。

但原版场景图里，这几条信号画成了：

- 从“控制”发往“模块”

这样一来，模型上就会出现一个解释冲突：

- 一边说这是驱动控制器内部状态变化的触发信号
- 另一边又把它画成控制器自己向外发出的输出

`v2` 对这三条信号做的修正是：

- `Sig9` 改为“模块 -> 控制”
- `Sig8` 改为“模块 -> 控制”
- `Sig7` 改为“模块 -> 控制”

这个修改的本质不是图形方向微调，而是把这三条信号的 **模型角色** 改正确了：

- 原版里，它们看起来像“控制器输出命令”
- `v2` 里，它们被明确成“外部/模块送给控制器的触发输入”

这样以后再解释这些迁移时，就可以统一说：

- `Sig9` 是外部触发 `H` 内部状态切换的输入
- `Sig8` 是外部触发 `H -> G` 的输入
- `Sig7` 是外部触发 `S -> O` 的输入

状态机和场景图的含义就一致了。

## 4. 用一句人话概括 `v2` 相比原版到底改了什么

如果是给不写代码、只在图形界面里建模的人沟通，可以直接这么说：

> `v2` 没有推翻原来的状态机结构，主要是把原来两处不够落地的判断条件改成了“能在场景里看见、能用具体数值解释”的规则，并且补了一条新的观测量 `z`。另外，把 `Sig9`、`Sig8`、`Sig7` 这三条原来方向画反的触发信号纠正过来，让时序图和状态机对同一件事的解释一致。

如果再展开一点，可以这样说：

1. 原来 `A -> B` 用的是 `a<b`，`v2` 改成了 `y<2200`，这样可以直接用场景中的 `y=2300`、`y=2099` 来解释为什么会切换。
2. 原来 `F -> W` 用的是 `c<d`，`v2` 改成了 `z<1200`，并新增了 `z` 的一组场景数值，让这条切换也有明确依据。
3. 原来 `Sig9`、`Sig8`、`Sig7` 在时序图里画成控制发给模块，`v2` 改成模块发给控制，这样它们才真正符合“触发控制状态切换的输入信号”这个角色。

## 5. 从模型质量角度看，`v2` 比原版多解决了什么问题

从建模质量角度，`v2` 主要额外解决了下面三类问题：

- **可解释性更强**
  原版的 `a<b`、`c<d` 很难和场景中的具体时刻对应。`v2` 改成阈值后，可以直接解释“为什么此时切换”。
- **可验证性更强**
  条件和场景观测量对上之后，更容易做时序一致性检查、状态可达性检查，以及后续自动转换。
- **状态机与场景的一致性更强**
  原版有一部分信号在状态机里像输入、在时序图里像输出。`v2` 把这类角色冲突修掉了。

## 6. 最终结论

`v2` 相比原版，本质上做的是一次 **模型语义校正**，不是一次 **模型结构重做**。

最重要的修改只有三类：

1. 把 `A -> B` 的条件从 `a<b` 改成 `y<2200`
2. 把 `F -> W` 的条件从 `c<d` 改成 `z<1200`，并补上 `z` 的场景轨迹
3. 把 `Sig9`、`Sig8`、`Sig7` 改成由模块发往控制，使它们与状态机里的“触发输入”角色一致

如果需要对外沟通，这三点就是最应该强调的内容。
