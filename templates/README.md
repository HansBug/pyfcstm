# Built-in template system

This directory contains the editable source for built-in `pyfcstm` templates. It is the maintainer-facing source of truth for template development, while `pyfcstm/template/` contains packaged zip assets and `pyfcstm/template/index.json` used by installed distributions.

The built-in template system has two responsibilities:

1. turn an FCSTM model into target-language files through the renderer; and
2. package selected repository template sources so `pyfcstm generate --template <name>` can use them without requiring a checkout of this repository.

Generated runtime semantics must stay aligned with the FCSTM model and simulator contracts. For generated implementation files, performance and semantic correctness have priority over human long-term maintainability. Formatter and linter gates exist to keep generated output professional and integration-friendly, not to make implementation readability more important than runtime behavior.

## Directory roles

| Path | Role | Maintainer notes |
| --- | --- | --- |
| `templates/README.md` | English template system handbook | Describes repository-wide template rules and maintenance contracts. |
| `templates/README_zh.md` | Chinese template system handbook | Must stay equivalent to this file for repository-wide rules. |
| `templates/<name>/` | Editable source for one built-in template | Only first-level directories are packaged as built-in templates. |
| `templates/<name>/README.md` | English maintainer handbook for one template | Describes that template's target, compatibility, API surface, maintenance flow, and tests. |
| `templates/<name>/README_zh.md` | Chinese maintainer handbook for one template | Must stay equivalent to the English template-level handbook. |
| `templates/<name>/README.md.j2` | English generated-output user guide | Rendered into the generated directory for downstream users. |
| `templates/<name>/README_zh.md.j2` | Chinese generated-output user guide | Rendered into the generated directory for downstream users. |
| `templates/<name>/machine.*.j2` | Target-language source templates | Rendered into generated runtime source files. |
| `templates/<name>/config.yaml` | Renderer configuration contract | Defines expression styles, statement styles, extension points, and ignore rules. |
| `templates/<name>/template.json` | Built-in template metadata | Describes the packaged template for `pyfcstm/template/index.json`. |
| `pyfcstm/template/` | Packaged built-in template assets | Contains zip archives and `index.json`; generated by `tools.package_templates`. |

Root `templates/README*.md` files are not part of any template archive. Changing only these root handbooks does not require rebuilding template zip assets.

## Current built-in templates

| Template name | `template.json.language` | Purpose |
| --- | --- | --- |
| `python` | `python` | Self-contained Python runtime template. |
| `c` | `c` | Self-contained C99 runtime template with C++98 integration support. |
| `c_poll` | `c` | Self-contained C99/C++98 runtime template using hook-polled events. |

Template name and target language are related but not identical. For example, `c_poll` is a distinct template whose generated target language is still `c`.

## Reserved language vocabulary

Future templates are expected to use stable `template.json.language` values. The following values are reserved for upcoming template work but are not required to exist as template directories now:

| Future language | Reserved value | Current expectation |
| --- | --- | --- |
| Java | `java` | Leave room for Java source banners, formatter/build gates, and standard-library-only runtime design. |
| JavaScript | `js` | Leave room for JavaScript source banners, formatter gates, and language-core/runtime-free design. |
| Rust | `rust` | Leave room for Rust source banners, `rustfmt`, and standard-library-only runtime design. |
| Ruby | `ruby` | Leave room for Ruby source banners and dependency-free runtime design. |
| Go | `go` | Leave room for Go source banners, `gofmt`, and standard-library-only runtime design. |

Checks should distinguish current templates from reserved vocabulary. `python`, `c`, and `c_poll` are current template names. `java`, `js`, `rust`, `ruby`, and `go` are reserved target-language vocabulary and must not force empty directories to exist.

All future formatter or linter gates for these reserved languages inherit the same repository-wide standard: they are pragmatic quality gates for obvious generated-code roughness, not absolute style objectives. A future Java, JavaScript, Rust, Ruby, Go, or other template must document its formatter flow with that constraint before adding strict checks.

## Renderer contract

The renderer is implemented primarily in `pyfcstm/render/render.py` and `pyfcstm/render/env.py`.

A template directory is consumed by `pyfcstm.render.StateMachineCodeRenderer` with these rules:

- `config.yaml` is loaded before file mappings are prepared.
- Jinja2 files ending in `.j2` are rendered, and the generated output path removes the `.j2` suffix.
- Non-`.j2` files are copied as static files, unless ignored.
- `config.yaml` itself is not copied to the generated output.
- Ignore rules use Git-style patterns through `pathspec` and are controlled by the template's `ignores` entry.
- The current render context passes `model` to templates. It does not pass the raw input source text.
- Expression rendering and statement rendering are exposed through filters such as `expr_render`, `stmt_render`, and `stmts_render`.

The core rendering call currently renders templates as `tp.render(model=model)`. Template authors must not assume additional context variables unless they are added deliberately to the renderer and documented here, in template-level README files, and in tests.

## `config.yaml` contract

`config.yaml` is the renderer configuration file for a template. Its known top-level keys are defined by `pyfcstm/render/render.py`, `pyfcstm/render/env.py`, `pyfcstm/render/expr.py`, `pyfcstm/render/statement.py`, and `pyfcstm/render/func.py`.

| Key | Meaning | Runtime boundary |
| --- | --- | --- |
| `expr_styles` | Adds or overrides expression rendering styles. | Generation-time only. |
| `stmt_styles` | Adds or overrides statement rendering styles. | Generation-time only. |
| `globals` | Adds Jinja2 global objects through declarative config items. | Generation-time only. |
| `filters` | Adds Jinja2 filters through declarative config items. | Generation-time only. |
| `tests` | Adds Jinja2 tests through declarative config items. | Generation-time only. |
| `ignores` | Git-style patterns excluded from rendering/copying. | Generation-time only. |

Most keys are optional and default to an empty configuration. Generation-time dependencies such as Jinja2, PyYAML, `pathspec`, renderer helpers, and imported filters are allowed because they run inside `pyfcstm` during generation. They must not leak into generated runtime dependencies unless the generated target language explicitly owns that dependency and it is approved by the runtime policy below.

When a new `config.yaml` key is added in code, update this handbook, the affected template-level README files, and the structure checks together.

## `template.json` metadata contract

Each packaged template directory should include `template.json`. The metadata loader is implemented in `tools.package_templates`.

| Field | Meaning |
| --- | --- |
| `name` | Template name. The packager overwrites it with the first-level directory name. |
| `title` | Human-readable template title. |
| `description` | Short description used by template discovery and documentation. |
| `language` | Target language of generated code, not necessarily the template name. |
| `experimental` | Whether the template is experimental. |
| `archive` | Packaged archive filename. The packager writes this as `<name>.zip`. |
| `root_dir` | Root directory inside the archive. The packager writes this as `<name>`. |

`name`, `archive`, and `root_dir` are authoritative after packaging because `tools.package_templates` forces them from the directory name. Template authors should still keep `template.json` readable and accurate for source review.

## Packaging and loading contract

The built-in packaging chain has several distinct authority points:

- `tools.package_templates` scans only first-level directories under `templates/`.
- `tools.package_templates` reads each template's `template.json`, merges it with default metadata, then forces `name`, `archive`, and `root_dir`.
- Before writing new archives, `tools.package_templates` removes stale `.zip` files from `pyfcstm/template/`.
- Each archive uses the template name as the zip root directory.
- `tools.package_templates` writes `pyfcstm/template/index.json` with the packaged template metadata.
- `setup.py` calls `package_templates()` when the repository checkout contains `templates/`.
- `MANIFEST.in` includes `pyfcstm/template/*.zip` and `pyfcstm/template/*.json` in source distributions.
- `setup.py` `package_data` includes `*.zip` and `*.json` package resources for wheels.
- `pyfcstm/template/__init__.py` provides `list_templates`, `has_template`, `get_template_info`, and `extract_template`.
- In an installed package, `extract_template` normally unpacks the zip archive referenced by `index.json`.
- In a development checkout, if the zip archive is absent, `extract_template` uses fallback copy from `templates/<name>`.
- `pyfcstm/entry/generate.py` extracts a built-in template for `pyfcstm generate --template <name>` and then hands the extracted directory to `StateMachineCodeRenderer`.

Do not replace this chain with ad-hoc runtime discovery. If packaging behavior changes, update `tools.package_templates`, `setup.py`, `MANIFEST.in`, `pyfcstm/template/__init__.py`, this handbook, and template packaging tests together.

## Documentation layering rules

Documentation must stay layered so template maintainers and generated-code users do not receive the wrong information.

| Layer | Audience | Should contain | Should avoid |
| --- | --- | --- | --- |
| Root `templates/README.md` / `templates/README_zh.md` | Template-system maintainers | Repository-wide mechanisms, governance, packaging, renderer contracts, future language vocabulary. | Per-template user tutorials or generated runtime API details. |
| Template `templates/<name>/README.md` / `templates/<name>/README_zh.md` | Maintainers of one template | Target language, compatibility, public integration surface, performance strategy, source layout, maintenance commands, tests. | Generated-output onboarding that belongs in `README.md.j2`. |
| Generated `README.md.j2` / `README_zh.md.j2` | Users of one generated output directory | What was generated, model summary, variable/event/hook usage, cold start, hot start, complete minimal run flow. | Internal template packaging mechanics. |
| Source `.j2` files such as `machine.py.j2`, `machine.c.j2`, `machine.h.j2` | Renderer and generated runtime | Efficient target-language implementation and concise generated-file guidance. | Long template-system explanations. |

A person or LLM that only sees a generated directory should be able to use the generated runtime from the generated README without knowing how the repository template system works.

## Generated README authoring rules

Generated `README.md.j2` and `README_zh.md.j2` files are user-facing runtime
guides, not template-maintainer notes. Treat them as executable onboarding
contracts for people and LLMs that only have the generated directory.

When editing generated README templates:

- Render at least one moderately complex checked-in FCSTM fixture that includes
  persistent variables, events, nested states, lifecycle behavior, and abstract
  hooks when the target template supports them.
- From the generated directory alone, write a small downstream program by
  following the generated README. For compiled languages, compile and run that
  program with the generated sources. For interpreted languages, import and run
  the generated module from outside the repository test helpers.
- Verify the downstream program exercises cold start, at least one eventful
  cycle when events exist, state/variable inspection, and hook or event-check
  installation when the generated model declares those extension points.
- Keep snippets copy-adaptable. Do not reference an undefined callback,
  variable, event-check field, hook field, include, or import unless the snippet
  is explicitly marked as schematic and points to a complete example nearby.
- For `c_poll`, every runnable cycle example for an eventful model must explain
  that a complete `EventChecks` table is required before `cycle()`. Partial
  tables are a runtime error, so examples must not imply that setting one field
  is enough for every generated model.
- Use canonical / normalized model export wording for text generated from
  `model.to_ast_node()`. Do not call it raw or original source.
- Keep generated README files focused on the generated runtime: file layout,
  model text, model summary, variables, events, hooks, cold start, hot start,
  cycle input, current-state/variable inspection, errors, and a complete
  minimal flow. Renderer internals, packaging mechanics, and template
  governance belong in root or template-level maintainer handbooks.

If a generated README example cannot be used to produce a small working
consumer program after straightforward identifier substitution, fix the README
template before claiming the template change is complete.

## Generated runtime policy

Generated runtime files must be strictly self-contained by default:

- No dependency on `pyfcstm runtime`.
- No third-party runtime dependency.
- No dependency on unstable, platform-specific, or unusually new language/library features.
- Prefer target-language core features and long-lived standard library functionality.
- Preserve broad version and platform compatibility. Existing expectations include Python 3.7+ for `python`, C99 for C implementation, and C++98-compatible integration paths for C headers/harnesses.

Generated implementation files should be optimized for model semantics, predictable execution, and runtime performance. Human readability is secondary for implementation files such as `machine.py` and `machine.c`. Public integration surfaces such as `machine.h` should remain clear because they are read by downstream integrators.

Formatter and linter checks are still required where defined. Their purpose is to make generated output look professional and avoid obvious integration friction. This is a pragmatic quality gate, not an unlimited pursuit of style perfection. Do not contort generated runtime design solely to satisfy a stylistic preference when it would harm performance, semantics, compatibility, or simulator/template alignment. If a rare formatter-only exception is accepted, document it narrowly with the runtime and compatibility reason.

This policy applies equally to existing formatter flows and future ones. New templates must not define stricter formatter rules that force maintainers to over-optimize extreme edge cases unrelated to visible behavior or integration quality.

For target languages that manage memory or other resources explicitly, template maintainers must also treat resource lifetime as part of runtime correctness. Generated runtimes can be embedded in long-running control systems, so generated code must have a clear ownership model for allocations, initialization, hot start, cycle execution, hook/event-check registration, and destruction. C/C++ templates and future native or resource-owning templates should run representative sanitizer, leak-check, valgrind, or equivalent harnesses when runtime source templates change. If an existing leak or resource-lifetime issue is discovered outside the current change scope, record it with a reproducible harness instead of silently ignoring it.

## Source context contract

The current renderer passes the model object but does not pass raw source text. Therefore generated artifacts must label model text precisely:

- Use `original source` only for bytes/text that truly came from the user's input file.
- Use `canonical model export`, `normalized model export`, or equivalent wording for text produced from `model.to_ast_node()` or another model export path.
- Do not call canonical export the raw source.
- If a future renderer adds raw source context, update `pyfcstm/render/render.py`, `pyfcstm/entry/generate.py`, generated README templates, source metadata, and tests together.

This contract applies to generated README files, `DSL_SOURCE`-style constants, C helper functions such as `_dsl_source()`, docstrings, comments, and any public/generated metadata.

## New template checklist

Before adding a new built-in template directory, confirm all items below:

- Add `templates/<name>/template.json` with accurate `title`, `description`, `language`, and `experimental` values.
- Add `templates/<name>/config.yaml` and use only documented renderer keys unless the renderer is extended in the same change.
- Add template-level `README.md` and `README_zh.md` for maintainers.
- Add generated-output `README.md.j2` and `README_zh.md.j2` for generated-code users.
- Add generated source templates with generated-file banners near the first real output.
- Keep generated runtime self-contained: no `pyfcstm runtime`, no third-party runtime dependency, no unstable runtime dependency.
- Define target-language compatibility, formatter, build, and test gates; formatter gates must be documented as pragmatic quality checks, not absolute style goals.
- For templates in languages with explicit resource ownership, define resource-lifetime and leak-check expectations alongside build and runtime gates.
- Add representative generated-runtime tests and, when applicable, simulator alignment tests.
- Run packaging checks so `pyfcstm/template/index.json`, zip archives, and `extract_template` agree with source templates.
- Update this handbook only when the repository-wide contract changes; update the template-level README when only one template changes.

## Maintenance workflow

Use the smallest verification set that covers the change:

- Root handbook-only changes: run README structure review and self-check; `make tpl` is not needed.
- Template source changes under `templates/<name>/`: run `make tpl` and template-specific tests.
- Generated runtime source-template changes: generate representative outputs, run formatter/build/runtime checks, and verify generated README examples where applicable.
- Python public API or pydoc changes: run `make rst_auto` and review generated RST diffs.

For local iteration, `SKIP_SLOW_TESTS=1 make unittest` skips C/C++ native toolchain template tests while keeping Python template, simulator, model, DSL, render, and verify tests active. Do not claim a template change is complete until checks proportional to that template's runtime language have passed.
