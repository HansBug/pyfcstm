# UPPAAL 可验证性质说明

本文介绍 UPPAAL 中“可验证性质”是什么、能写成什么语法、其形式语义是什么，以及使用时最容易混淆的地方。默认讨论的是经典 UPPAAL `verifyta` 所使用的 **symbolic queries**。这也是大多数人在说“UPPAAL 性质表达式”时指的那一套。

需要先说明一点：UPPAAL 并不是一个“任意 CTL/TCTL 公式都能直接输入”的全功能时序逻辑证明器。它支持的是一组非常实用、经过约束的查询形式。这些查询足够表达可达性、安全性、活性、响应性，以及若干数值上界/下界分析问题。

## 1. 什么是 UPPAAL 的“可验证性质”

在 UPPAAL 中，系统模型通常是由若干个带时钟的自动机并行组合而成。验证器面对的是由以下信息构成的状态空间：

- 每个进程当前所处的位置
- 各个离散变量的当前取值
- 各个时钟的当前取值

查询语言的目标，就是对这个状态空间提出问题，例如：

- 某个坏状态是否可达
- 某个安全条件是否始终成立
- 某个目标状态是否必然最终出现
- 某个请求发生后，是否一定最终得到响应
- 某个变量或时钟在某些状态下的可能取值范围是多少

这些“问题”就是 UPPAAL 中的性质表达式，也常叫 query。

## 2. 两类常见性质语言

UPPAAL 家族里不止一种查询语言。最容易混淆的是下面两类：

- **经典 symbolic queries**
  - 对 timed automata 的符号状态空间做严格验证
  - 典型语法是 `A[] p`、`E<> p`、`p --> q`
- **统计模型检验 SMC queries**
  - 用采样和概率估计做统计检验
  - 典型语法是 `Pr[<=10](<> goal)`、`E[<=20;100](max: cost)`

本文重点是第一类，也就是经典 symbolic queries；最后会补充说明 SMC 与它的区别。

## 3. 经典 symbolic queries 的核心语法

UPPAAL 官方文档中，经典查询的核心形式可以概括为：

```text
A[] φ
E<> φ
E[] φ
A<> φ
φ --> ψ

sup: e1, e2, ...
sup{φ}: e1, e2, ...
inf: e1, e2, ...
inf{φ}: e1, e2, ...
bounds: e1, e2, ...
bounds{φ}: e1, e2, ...
```

其中：

- `φ`、`ψ` 是状态谓词，也就是“在某个状态上判真假的表达式”
- `A` 表示“对所有路径”
- `E` 表示“存在一条路径”
- `[]` 表示“始终”
- `<>` 表示“最终”
- `-->` 表示“leads-to”，也可理解为响应性质

从用途上看，上面这些查询可以分成两组：

- **布尔型性质**
  - `A[] φ`
  - `E<> φ`
  - `E[] φ`
  - `A<> φ`
  - `φ --> ψ`
- **数值型性质**
  - `sup`
  - `inf`
  - `bounds`

前者回答“真还是假”，后者回答“上界/下界/区间是多少”。

## 4. 状态谓词 `φ` 能写什么

时序操作符外面的 `φ`、`ψ` 本质上是状态表达式。它们是在某一个系统状态上求值的，因此只能依赖当前状态中的信息，而不能有副作用。

常见可写内容包括：

- 位置谓词：`P.Idle`、`P.Critical`
- 特殊谓词：`deadlock`
- 布尔表达式：`a && b`、`not error`
- 整数比较：`count < 3`
- 时钟约束：`x <= 5`、`x - y < 2`
- 逻辑蕴含：`p imply q`
- 量词：`forall (...)`、`exists (...)`
- 聚合：`sum (...)`

典型示例：

```text
P.Done
not deadlock
P.CS imply not Q.CS
x <= 5 and y - x < 3
forall(i : int[0, 3]) not worker[i].Error
exists(i : id_t) req[i]
```

### 4.1 位置谓词

如果模板实例名为 `P`，位置名为 `Wait`，那么：

```text
P.Wait
```

表示“进程 `P` 当前位于位置 `Wait`”。

这类表达式是 UPPAAL 查询里最常见的原子命题。

### 4.2 `deadlock`

`deadlock` 是一个特殊关键字，表示当前状态处于死锁。通常最常见的查询之一是：

```text
A[] not deadlock
```

它表示系统所有可达状态都不是死锁状态。

需要注意，UPPAAL 里的 `deadlock` 不只是“当前立刻没有离散边能走”，它的语义更强：系统既不能立即执行动作，也不能通过等待一段时间后再执行动作。

### 4.3 无副作用限制

查询中的状态表达式必须是 **side-effect free** 的。也就是说，你可以比较、计算、组合布尔条件，但不能在 query 里修改变量、更新时钟或触发状态变化。

因此下面这种“更新式”思路不属于 query 语言的合法用法：

```text
i = i + 1
```

查询是“观察”状态，而不是“执行”程序。

## 5. 五类经典布尔性质的语法与语义

这一部分是理解 UPPAAL 性质语言的核心。

### 5.1 `E<> φ`：可达性

语法：

```text
E<> φ
```

直观含义：

- 存在一条执行路径
- 在这条路径上的某个未来状态中，`φ` 成立

因此它表示：**`φ` 是可达的**。

例子：

```text
E<> P.Done
E<> P.CS and Q.CS
E<> error
```

解释：

- `E<> P.Done`：存在某次执行，最终能到达 `P.Done`
- `E<> P.CS and Q.CS`：存在某条路径，使两个进程同时进入临界区
- `E<> error`：错误状态是否可达

这是找反例、找坏状态、找可达目标时最常用的查询。

### 5.2 `A[] φ`：不变性/安全性

语法：

```text
A[] φ
```

直观含义：

- 对所有可能路径
- 在所有可达状态上
- `φ` 都成立

因此它表示：**`φ` 始终成立**，或者说 **坏事永远不会发生**。

例子：

```text
A[] not deadlock
A[] not (P.CS and Q.CS)
A[] x <= 10
```

解释：

- `A[] not deadlock`：系统永不死锁
- `A[] not (P.CS and Q.CS)`：互斥始终成立
- `A[] x <= 10`：时钟或变量 `x` 在所有可达状态中都不超过 10

这是最典型的安全性验证形式。

### 5.3 `A<> φ`：必然最终到达

语法：

```text
A<> φ
```

直观含义：

- 对所有路径
- 最终都能到达一个满足 `φ` 的状态

因此它表示：**无论系统怎样演化，`φ` 最终都会发生**。

例子：

```text
A<> P.Done
A<> all_finished
```

这类查询常用于表达“终止性”“必然完成”“必然恢复”等性质。

要特别区分它和 `E<> φ`：

- `E<> φ` 只要求“至少有一条路径能到达”
- `A<> φ` 要求“所有路径都最终到达”

前者是“可能”，后者是“必然”。

### 5.4 `E[] φ`：存在一条始终满足 `φ` 的路径

语法：

```text
E[] φ
```

直观含义：

- 存在一条路径
- 这条路径上的每个状态都满足 `φ`

因此它表示：**系统存在某种演化方式，可以一直保持 `φ`**。

例子：

```text
E[] not P.Error
E[] waiting
```

这类性质不如 `A[]` 和 `E<>` 常用，但它在表达“存在一种一直维持某条件的执行方式”时很有用。

最容易出错的地方是把它看成“所有路径都始终满足”。那其实是 `A[] φ`，不是 `E[] φ`。

### 5.5 `φ --> ψ`：响应性质 / leads-to

语法：

```text
φ --> ψ
```

直观含义：

- 每当某个状态满足 `φ`
- 之后都必须最终到达某个满足 `ψ` 的状态

因此它表示：**`φ` 发生后，`ψ` 一定最终跟上来**。

例子：

```text
request --> grant
P.Fault --> P.Recover
P.Waiting --> P.Running
```

这类性质特别适合表达：

- 请求最终得到响应
- 故障最终得到恢复
- 等待最终获得服务

要点是“每次前件出现后，都必须保证将来出现后件”，而不是“某次前件之后随便有一次后件就行”。

## 6. 这些性质的形式语义

为了更准确地理解它们，需要知道 UPPAAL 在语义上把系统看成一个 timed transition system。状态之间有两类迁移：

- **离散迁移**：执行边、同步、更新
- **时间迁移**：时间流逝一段非负实数长度

一条路径就是这两类迁移交替构成的执行序列。

在这个框架下，可以用更形式化的方式理解上述查询：

- `E<> φ`
  - 存在一条从初始状态出发的路径
  - 在这条路径的某个状态上 `φ` 为真
- `A[] φ`
  - 对所有从初始状态出发的路径
  - 其上所有状态都满足 `φ`
- `A<> φ`
  - 对所有从初始状态出发的路径
  - 总存在将来的某个状态满足 `φ`
- `E[] φ`
  - 存在一条从初始状态出发的路径
  - 这条路径上所有状态都满足 `φ`
- `φ --> ψ`
  - 对所有可达状态，只要该状态满足 `φ`
  - 则从该状态继续出发，所有后续演化都必须最终满足 `ψ`

从逻辑类别上看，它们分别对应：

- `E<>`：可达性
- `A[]`：安全性/不变性
- `A<>`：活性/终止性
- `E[]`：存在性保持
- `-->`：响应性

## 7. 常用等价关系

UPPAAL 官方文档给出了一些很重要的等价关系：

```text
A[] φ    ≡ not E<> not φ
A<> φ    ≡ not E[] not φ
φ --> ψ  ≡ A[] (φ imply A<> ψ)
```

这些等价关系有两个用途：

- 帮助理解不同查询的语义联系
- 在脑中把问题转化为“找反例”或“找可达坏状态”

例如，验证：

```text
A[] safe
```

等价于检查：

```text
not E<> not safe
```

也就是“是否不存在一个可达坏状态”。

不过要注意：虽然语义上有这些等价关系，UPPAAL 的输入语言并不意味着允许你自由书写任意嵌套时序公式。通常你应直接使用它支持的固定查询模式。

## 8. 状态表达式的语法要点

### 8.1 逻辑连接词

常用逻辑连接词包括：

- `not`
- `and`
- `or`
- `imply`

例如：

```text
A[] (P.CS imply not Q.CS)
A[] (not error and x <= 5)
```

### 8.2 量词

如果模型里有数组、标识符类型或索引范围，query 中常会使用量词：

```text
forall(i : int[0, 3]) not worker[i].Error
exists(i : id_t) req[i]
```

这让你可以写出更规整的全局性质，而不需要手工展开每一个进程实例。

### 8.3 时钟与差分约束

查询中的状态谓词可以引用时钟值，也可以使用差分形式：

```text
x <= 5
x - y < 3
```

相比某些建模位置上的 guard/invariant 语法限制，query 中的布尔组合通常更灵活。

### 8.4 布尔与整数

UPPAAL 的表达式系统中，整数和布尔之间有一定兼容性。很多场景中 `0` 被视为假、非 `0` 被视为真，但在写正式 query 时，更推荐写成显式布尔条件，避免语义歧义。

## 9. `deadlock` 的精确定义与误区

`deadlock` 是 UPPAAL 查询里非常重要的一个概念，但也是最容易被直觉误解的一个。

很多人的直觉是：

- “当前没有边能走了，就是死锁”

但在 timed automata 中还存在“等时间再走”的可能。因此 UPPAAL 中 `deadlock` 更准确地表示：

- 当前状态既不能立刻执行离散动作
- 也不能通过让时间继续流逝而到达一个可执行动作的状态

因此下面这条是标准安全检查：

```text
A[] not deadlock
```

如果它不成立，说明存在某个可达状态，系统彻底卡住了。

## 10. `A[]` 检查的到底是什么

还有一个常见误区是对 `A[] φ` 的理解过于“程序执行过程化”。

UPPAAL 中状态谓词是在“状态”上求值的，而不是在某次同步/更新动作的中间瞬间逐步求值。也就是说，验证器关心的是：

- 初始状态是否满足
- 每个可达后继状态是否满足

它不等价于“每条边更新过程中每一个中间子步骤都必须满足”。

这在同步边上带多个更新、或者多个组件同步跳转时尤其需要注意。你验证的是状态空间中的离散状态点，而不是实现层面的逐条赋值指令序列。

## 11. 数值查询：`sup`、`inf`、`bounds`

除了真假型性质，UPPAAL 还支持数值分析查询。

### 11.1 `sup`

语法：

```text
sup: e
sup{φ}: e
```

含义：

- 计算表达式 `e` 的上界或上确界
- 如果写成 `sup{φ}: e`，则只在满足 `φ` 的状态中考虑

例子：

```text
sup: x
sup{P.Busy}: cost
```

可理解为：

- `x` 最多能到多大
- 在 `P.Busy` 状态下，`cost` 的最大可能值是多少

### 11.2 `inf`

语法：

```text
inf: e
inf{φ}: e
```

含义：

- 计算表达式 `e` 的下界或下确界
- 若带过滤条件 `{φ}`，则只在满足 `φ` 的状态中统计

例子：

```text
inf: t
inf{P.Wait}: x
```

### 11.3 `bounds`

语法：

```text
bounds: e
bounds{φ}: e
```

含义：

- 计算表达式 `e` 的可能取值边界集合
- 带过滤条件时，只在满足 `φ` 的状态中统计

例子：

```text
bounds: x
bounds{P.Run}: x
```

这类查询适合回答：

- 某个时钟的可取值范围是什么
- 某个变量在某位置上可能落在哪些区间

## 12. 典型性质模板

下面列出一些最常见的建模需求以及对应的 query 模板。

### 12.1 坏状态不可达

```text
A[] not bad
```

或者从反例视角写成：

```text
E<> bad
```

前者表示“坏状态永不可达”，后者表示“坏状态是否可达”。

### 12.2 互斥

```text
A[] not (P.CS and Q.CS)
```

如果有多个进程，也可以配合量词写更一般的形式。

### 12.3 无死锁

```text
A[] not deadlock
```

### 12.4 终止性

```text
A<> done
```

表示所有执行最终都会达到完成状态。

### 12.5 响应性

```text
request --> grant
```

如果请求可能由多个进程提出，也可用量词组合出更一般的响应性质。

### 12.6 资源上界

```text
sup: cost
sup{Busy}: waiting_time
```

## 13. 经典查询语言的能力边界

虽然 UPPAAL 的 symbolic query 语言很强大，但它并不是“无限制逻辑语言”。使用时要明确以下边界：

- 它不是任意 CTL/TCTL 公式输入器
- 一般不应假定支持任意深度的时序算子嵌套
- 它的重点是若干固定但高频、实用的验证模式

因此在表述需求时，最好尽量先把问题归类为：

- 可达性
- 安全性
- 活性
- 响应性
- 数值界分析

然后再映射成 UPPAAL 的支持语法。

## 14. 与 SMC 查询的区别

如果你使用的是 UPPAAL SMC，那么性质语言会明显不同。例如：

```text
Pr[<=10](<> goal)
E[<=20](max: cost)
```

这类查询的特点是：

- 面向概率或统计估计
- 对随机仿真轨迹进行采样
- 返回概率估计、置信区间或期望值估计

而经典 symbolic queries 的特点是：

- 不依赖统计采样
- 在符号状态空间上做严格验证
- 返回真/假，或确切的上下界信息

因此在写“UPPAAL 性质”时，首先要确认自己用的是哪一个子工具。

## 15. 实践建议

### 15.1 先写反例查询，再写全局性质

例如想验证互斥：

```text
A[] not (P.CS and Q.CS)
```

调试模型时往往可以先写：

```text
E<> P.CS and Q.CS
```

这样如果坏状态可达，验证器通常更容易给出直观反例。

### 15.2 区分“可能”与“必然”

建模时最常见的逻辑错误，就是把下面两件事混淆：

- `E<> φ`：某件事有可能发生
- `A<> φ`：某件事一定会发生

前者成立并不意味着后者成立。

### 15.3 区分“所有路径始终”与“存在路径始终”

同样要严格区分：

- `A[] φ`
- `E[] φ`

它们只差一个量词，但语义强度完全不同。

### 15.4 对响应性质优先用 `-->`

如果你要表达“请求之后最终响应”“故障之后最终恢复”，优先直接用：

```text
φ --> ψ
```

它比手工在脑中翻译成嵌套时序表达更不容易出错。

## 16. 一页速查

```text
E<> φ      存在一条路径，最终到达 φ
A[] φ      所有路径上的所有状态都满足 φ
A<> φ      所有路径最终都会到达 φ
E[] φ      存在一条路径，其上始终满足 φ
φ --> ψ    每次 φ 发生后，之后最终一定出现 ψ
sup: e     求 e 的上界
inf: e     求 e 的下界
bounds: e  求 e 的可能边界/区间
```
