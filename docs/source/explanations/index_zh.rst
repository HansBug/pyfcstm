解释地图
================

当你的问题是“为什么 pyfcstm 会这样工作”时，从本页开始。全站左侧导航会直接列出每个解释页面；
本地图帮助你选择负责机制、顺序、架构边界或设计取舍的同级页面。

页面角色：本页是解释（explanation）入口，负责把概念问题分流到架构、语言语义、执行语义、诊断、
模板渲染和语法工具链页面，而不是把解释写成教程或查表页。

不负责内容：本页不提供首次成功命令、任务步骤、闭合集合选项表、JSON 模式、诊断码表或应用程序接口签名。
学习路径看 :doc:`/tutorials/index_zh`，
任务步骤看 :doc:`/how_to/index_zh`，
精确事实看 :doc:`/reference/index_zh`。

解释承诺什么
------------------

解释应该回答某个行为为什么存在、系统按什么顺序执行、哪个子系统拥有哪个边界，以及某个设计取舍的原因。解释可以使用轨迹、
图或例子，但例子服务于推理，而不是复制粘贴任务。需要精确事实时，解释应该链接到参考页，而不是复制完整表格。

新用户路径
----------------

如果你从教程进入概念理解，可以按下面顺序读：

1. 先读 :doc:`architecture/index_zh`，看从语言文本到模型、诊断、渲染、
   仿真和生成产物的流水线。
2. 当语法例子不够时，读 :doc:`dsl_semantics/index_zh`，理解模型含义。
3. 当周期、生命周期动作或转换轨迹让你意外时，读 :doc:`execution_semantics/index_zh`。
4. 当检查输出需要解释时，读 :doc:`diagnostics/index_zh`。
5. 当生成代码行为依赖渲染配置或打包模板边界时，读 :doc:`template_rendering/index_zh`。
6. 只有在维护语法、高亮、编辑器资产或面向大语言模型的语法材料时，
   才读 :doc:`grammar_tooling/index_zh`。

有经验用户路径
---------------------

如果你已经知道子系统，按问题选择：

* 这个行为属于包流水线的哪个层面？看 :doc:`architecture/index_zh`。
* 这个语言构造在解析和导入之后是什么意思？看 :doc:`dsl_semantics/index_zh`。
* 为什么某次周期产生这个活跃状态和变量轨迹？看 :doc:`execution_semantics/index_zh`。
* 检查能证明什么、警告什么、只能描述什么？看 :doc:`diagnostics/index_zh`。
* 为什么生成流程要经过模板、表达式样式和语句样式？看 :doc:`template_rendering/index_zh`。
* 为什么一次语法修改会影响解析器、高亮、文档和编辑器？看 :doc:`grammar_tooling/index_zh`。

维护者路径
----------------

审查解释质量时，检查下面几点：

* 每篇解释是否回答“为什么”或“怎样工作”，而不是重复命令表。
* 复杂行为是否有轨迹、图或边界例子，避免只靠抽象文字描述。
* 风险范围是否准确，例如 C/C++ 生成部署警告不能写成 Python 运行时警告。
* 中英文页面是否讲同一机制，并且没有在某一语言中丢掉图、轨迹或边界说明。

解释阅读信号
------------------

读解释页时，不应该只收集术语。
你应该能把一个现象放到更大的因果链里：
输入是什么、系统先做什么、哪个边界会停止、
哪些结果只是警告，哪些结果能证明问题存在。

如果你需要的是下一条命令，
说明你已经离开了解释问题，应回到任务指南。
如果你需要的是一个合法值、默认值或字段名，
说明你需要参考页。
如果你还不知道从哪个现象开始，
可以先回到教程获得一个更小的可观察例子。

解释页的价值在于让后续任务和参考更容易使用，
而不是把它们全部复制到同一页。
这也是维护者审查解释页时最重要的边界。
如果一段解释无法指出自己改善了哪个后续判断，
它很可能只是普通目录文字，需要继续加厚。
解释页可以很长，但每一段都应该服务于一个后续判断，
而不是只增加术语密度。

概念卡片
----------------

架构：:doc:`architecture/index_zh`
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

读前条件：你知道 pyfcstm 有源文件、模型对象、命令、模板、诊断、仿真和生成输出，但需要理解这些部分怎样连接。

读后能力：你能把一个行为放到语言解析器、模型导入、检查、渲染、仿真、验证、可视化或生成资产边界中。

不负责内容：它不教命令使用，也不定义每个类和函数。

下一步：先用架构轨迹选择更深解释；需要精确命令或接口事实时再看参考。

语言语义：:doc:`dsl_semantics/index_zh`
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

读前条件：你能读基本 FCSTM 语法，并需要理解状态、转换、事件作用域、生命周期钩子、切面、
伪状态或组合形式在解析后的含义。

读后能力：你能解释所有权、名称解析、组合状态进入、转换形式、表达式分离、生命周期复用和语法糖展开。

不负责内容：它不替代完整语法参考，也不替代模型编辑任务指南。

下一步：精确形式查 :doc:`/reference/dsl/index_zh`；
写模型任务看 :doc:`/how_to/dsl/index_zh`。

执行语义：:doc:`execution_semantics/index_zh`
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

读前条件：你有合法模型或仿真轨迹，并需要推理周期、进入、期间动作、退出、转换效果、热启动、回滚或仿真器对齐。

读后能力：你能解释为什么一次周期产生某个活跃路径，以及为什么某些生命周期或切面动作按特定顺序运行。

不负责内容：它不列出所有仿真命令或所有生成运行时接口字段。

下一步：操作任务看 :doc:`/how_to/simulation/index_zh`；
精确命令和接口事实看 :doc:`/reference/simulation/index_zh`。

有界模型检查：:doc:`bmc_semantics/index_zh`、
:doc:`bmc_properties/index_zh` 与 :doc:`bmc_solving/index_zh`
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

读前条件：你能读 FCSTM 模型和 FBMCQ 查询，但需要理解有界结论、性质极性、见证与重放
为何具有当前含义。

读后能力：三页依次解释轨迹符号与核心转换关系、定义性与七类性质目标，以及双重求解、
结论映射、见证投影和运行时重放。

不负责内容：它们不教授首次使用，也不充当字段目录。

下一步：操作任务看 :doc:`/how_to/bmc/index_zh`；精确语法与协议事实查
:doc:`/reference/bmc_query/index_zh` 和 :doc:`/reference/bmc_results/index_zh`。

诊断：:doc:`diagnostics/index_zh`
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

读前条件：你有检查输出、诊断密集模型，或想知道静态分析能看见什么、不能证明什么。

读后能力：你能区分已解析结构、静态诊断、可选验证反馈、输出层级、严重级别和大语言模型修复价值。

不负责内容：它不枚举每个 JSON 报告字段或每个诊断码。

下一步：模式细节查 :doc:`/reference/inspect_report/index_zh`；
诊断码查 :doc:`/reference/diagnostics_codes/index_zh`；
排查步骤看 :doc:`/how_to/inspect/index_zh`。

模板渲染：:doc:`template_rendering/index_zh`
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

读前条件：你正在看生成输出或自定义模板，并需要理解为什么渲染经过模板目录、打包资产、Jinja 辅助对象、
表达式样式和语句样式。

读后能力：你能区分渲染器职责和目标运行时职责，并理解为什么内置模板要通过公共提取路径打包和测试。

不负责内容：它不列出每个模板配置键或每个内置目标契约。

下一步：模板任务看 :doc:`/how_to/templates/index_zh`；
配置事实查 :doc:`/reference/template_config/index_zh`；
内置目标事实查 :doc:`/reference/builtin_templates/index_zh`。

语法工具链：:doc:`grammar_tooling/index_zh`
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

读前条件：你准备修改语法或编辑器工具，并需要理解为什么解析语法、语义模型、高亮、文档、
大语言模型指南和 VSCode 资产要一起移动。

读后能力：你能解释语法解析、语义模型导入、语法高亮、编辑器补全和面向提示的语法指南之间的区别。

不负责内容：它不替代重新生成和验证资产的精确命令清单。

下一步：任务步骤看 :doc:`/how_to/grammar_editor/index_zh`；
规范文件和命令查 :doc:`/reference/grammar_tooling/index_zh`。

解释在哪里停止
---------------------

当问题变成操作或事实时，请离开解释。需要运行命令就去任务指南；需要合法形式、默认值、字段名或诊断码就去参考。
解释的成功标准是让下一个操作页或事实页更容易使用。
