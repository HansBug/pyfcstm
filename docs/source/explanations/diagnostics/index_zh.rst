诊断解释
========

Inspect 诊断用于回答三个问题：

1. 当前模型里有什么？
2. 哪些部分无效、有风险、未使用、不可达或对目标配置敏感？
3. 人、CI 作业、IDE 或 LLM 修复循环下一步应该使用哪些结构化上下文？

Inspect 能做什么
----------------

默认检查是对已解析模型运行的静态分析。它能看到状态层级、转换、事件、变量、动作、派生可达性、事件发射、变量数据流、切面影响、动作引用、组合来源和注册表支持的诊断。

这就是许多诊断带 ``span`` 和 ``refs`` 的原因。``span`` 指回源码；``refs`` 命名变量、状态、事件、动作、守卫、目标配置或来源对象，供修复循环继续检查。

默认 inspect 不能证明什么
-------------------------

默认 inspect 不是完整模型检查器。它不会探索每条运行时轨迹、每个外部事件来源、每个抽象处理器或每种目标语言部署，也不会自动运行无界验证算法。

需要有界验证支持的诊断时，显式使用 ``--enable-verify``。即便如此，求解器支持的检查也可能受超时、求解器未知结果和检查策略边界影响。

严重级别不等于风险等级
----------------------

``error`` 表示 pyfcstm 不能安全构建或使用请求的模型。``warning`` 表示模型仍可检查，但某个事实可疑、有风险、死亡、冗余或对目标配置敏感。``info`` 表示分析器发现了非阻塞事实，通常帮助读者判断模型是否有意如此。

警告不自动等于发布阻塞项，但如果出现在已检查的教程或回归夹具中，就必须解释。

为什么发射层级重要
------------------

注册表包含的内容多于默认命令行输出：

* ``static_pipeline`` 诊断可以出现在普通 inspect 输出中。
* ``verify_pipeline`` 诊断需要可选验证集成。
* ``lookup_api`` 诊断描述解析 API 失败，不是默认静态输出。
* ``partial_static_pipeline`` 诊断可能只在某一端实现中发射。
* ``catalog_only`` 诊断保留兼容契约，即使当前正常路径不应发射。

如果参考页忽略这个区别，就会误导 CI 过滤和 LLM 修复循环。

为什么诊断能帮助 LLM
--------------------

LLM 修复提示如果消费结构化事实，而不是从散文里猜，会安全得多。一个有用诊断会告诉模型：

* 要查的稳定 ``code``；
* 用于排序优先级的 ``severity``；
* 需要编辑或检查的精确源码范围；
* 可用于分支处理的 ``refs`` 载荷；
* 注册表里的建议动作和禁止做法。

LLM 报告格式会直接暴露这些事实，但它们仍依赖和完整 JSON 诊断相同的注册表。
